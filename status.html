AutomateIt status

Beta

AutomateIt is feature-complete, exceeds the capabilities of most major products, and ensures its quality with a self-test suite. However, this is a young product and users are expected to be technically proficient, be willing to accept rough spots, work through problems and upgrade frequently.

Driver-related issues

Platform-specific drivers

Most AutomateIt features will work on all Ruby-supported platform. However, some plugins require platform-specific drivers. For example, AutomateIt’s package_manager.install command can use a driver to use Debian’ apt-get packaging tool.

Most plugins include drivers that provide limited features for many platforms. For example, the SYSV Service driver can start daemons on any platform using SystemV-style init—however, it cannot enable them to start after reboot because this requires a specialized driver, such as Chkconfig for RedHat-style systems.

AutomateIt’s production-quality release will provide complete, tested drivers for at least the platforms listed below.

For example, on the chart above, we can see that Ubuntu has been tested and has full-featured plugins for Account, Address, Package, Platform and Service. In contrast, Solaris hasn’t been tested yet, but should work with the limited-purpose drivers for Account, Address, Platform and Service.

Technical information about specific drivers can be found in the Documentation—for example the Service plugin driver for Chkconfig will be documented as AutomateIt::ServiceManager::Chkconfig.

Determine Driver availability

Run the integration test suite to see what drivers are supported on a platform. See the documentation’s TESTING.txt for details.

Writing drivers

If no driver is available, it’s easy to write one. Most drivers can be written in less than an hour. To write a driver, find the most similar driver available for a specific plugin, copy it, and rework its code. Save the code for the new driver in a file ending with .rb into the project’s lib directory, it will be automatically loaded whenever the Interpreter for that project is run. Please test and contribute drivers so that others can benefit.

Workarounds for missing drivers

If writing a driver seems like too much work, there are alternatives depending on what functionality is needed.

For example, AutomateIt adds tags based on the machine’s platform information—it will add an ubuntu tag when run on an Ubuntu system. For platforms that don’t have a Platform driver, users can manually set tags by defining them in the tags.yml file.

Or for example, AutomateIt provides drivers for installing software using platform-specific packaging tools like apt-get. If a user needs to use an unsupported package tool and doesn’t want to write a driver, they can manually use sh commands to run the system-level tool directly. This isn’t pretty, but will get the job done.

Future plugins

Plugins that might be implemented in the future:

  • ScheduleManager—Manipulates regular command execution schedules, e.g. cron.
  • TransferManager—Transfers files from some source, e.g. rsync, sftp, http.
  • SourceManager—Manipulates source control systems, e.g. svn, cvs, hg.
